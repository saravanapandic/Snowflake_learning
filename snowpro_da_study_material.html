Domain 1 — SQL Functions & Query Techniques

High Weight

Aggregate Functions

  • AVG / SUM / MIN / MAX / COUNT ignores NULLs
  • ANY_VALUE — returns an arbitrary non-null value from group (non-deterministic)
  • APPROX_COUNT_DISTINCT — fast HLL-based cardinality estimate
  • HLL() — HyperLogLog for huge datasets
  • LISTAGG, OBJECT_AGG, ARRAY_AGG

Statistical Functions

  • CORR(x, y) — Pearson correlation coefficient
  • COVAR_SAMP / COVAR_POP — covariance
  • REGR_SLOPE — linear regression slope (for forecasting)
  • REGR_INTERCEPT — y-intercept of regression line
  • VARIANCE / STDDEV — dispersion measures
  • APPROX_PERCENTILE — fast percentile on large sets

Window Functions

  • OVER (ORDER BY col ROWS BETWEEN n PRECEDING AND CURRENT ROW) — moving avg
  • ROW_NUMBER / RANK / DENSE_RANK
  • LAG / LEAD — access adjacent rows
  • FIRST_VALUE / LAST_VALUE
  • PERCENTILE_DISC(0.4) WITHIN GROUP (ORDER BY col DESC)

Math & Conversion

  • CEIL(x, scale) — round up to scale decimal places
  • ::FLOAT, ::INTEGER, ::VARCHAR — Snowflake cast syntax
  • 100::FLOAT * 20.78::INTEGER || ' Square Feet' — string concat with cast
  • TRUNC / ROUND / FLOOR

String Functions

  • ILIKE — case-insensitive LIKE
  • ILIKE 'p%_j%' ESCAPE '=' — escape special chars with ESCAPE clause
  • _ matches exactly one character; % matches zero or more
  • SPLIT_TO_TABLE(col, delimiter) — lateral flatten to rows
  • LATERAL STRTOK_SPLIT_TO_TABLE(col, '-') returns SEQ, INDEX, VALUE

Sampling

  • SAMPLE BLOCK (p) — block-level probability sampling
  • SAMPLE (n ROWS) — fixed row count sample
  • REPEATABLE (seed) — reproducible random sample
  • SELECT * FROM tbl SAMPLE (10 ROWS) — Snowflake syntax
  • select * FROM FCT_SALES SAMPLE (10 rows) — correct form
💡

Exam trap: CEIL(AVG(GRADE), 2) on grades {4,1,2,1,2,NULL} → AVG ignores NULL → AVG = 2.0 → CEIL(2.0, 2) = 2.00 (keeps 2 decimal places). Answer: A.

ANY_VALUE vs MIN/MAX Tricky

ANY_VALUE returns some value from the group — it's non-deterministic but never NULL if any non-null exists. Use it when you need a value but don't care which one (avoids expensive full grouping).

7-Day Moving Average Formula

SELECT AVG(SALES) OVER ( ORDER BY SALES_DATE ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS moving_avg_7d

1 record/day → 6 PRECEDING + CURRENT = 7 rows. Not SUM, not 7 PRECEDING.

Medium Weight

FULL OUTER JOINAll rows from BOTH tablesUnmatched on either side → NULL; use for "all projects AND all employees"

💡

Exam pattern: "Return ALL projects AND ALL employees" → FULL OUTER JOIN. If only all from one side → LEFT JOIN. If only matching → INNER JOIN.

INNER JOIN result on tbl_a × tbl_b Example

tbl_a IDs: {1,2,3,null} | tbl_b IDs: {1,1,2,5,null} → INNER JOIN on a.id=b.id → matches on 1,1,2 → 3 rows returned (IDs: 1,1,2). NULLs never match.

UNPIVOT Reshape

Converts wide columns into rows. A table with DEPT, JAN, FEB, MAR, APRIL → UNPIVOT produces rows with DEPT, MONTH, SALES. PIVOT does the reverse (rows → columns).

Domain 3 — Semi-Structured Data (JSON / VARIANT)

High Weight

JSON Path Navigation Syntax

-- Dot notation json_col:employee.name json_col:employee.phone[0] -- first item (no cast = variant) json_col:employee.phone[1]::varchar -- cast = removes quotes -- Array element access SELECT col_b[ARRAY_SIZE(col_b)-1] -- last element
  • Without cast → result keeps JSON quotes (string type VARIANT)
  • With ::varchar cast → quotes stripped

ARRAY Functions Key Functions

  • ARRAY_SIZE(arr) — number of elements
  • col_b[ARRAY_SIZE(col_b)-1] — last element (0-indexed)
  • col_b[-1] — does NOT work in Snowflake
  • LAST_VALUE(col_b) — window function, not array function
  • GET(arr, index) — alternative accessor
  • FLATTEN — lateral view exploding array into rows

INFER_SCHEMA Schema Detection

  • Detects schema from staged semi-structured files (Parquet, JSON, Avro etc.)
  • Use with CREATE TABLE … USING TEMPLATE (SELECT ARRAY_AGG(OBJECT_CONSTRUCT(*)) FROM TABLE(INFER_SCHEMA(…)))
  • NOT for unstructured data; NOT for JSON inside a table cell
  • Pair with GENERATE_COLUMN_DESCRIPTION to auto-create DDL

Loading Parquet from Stage COPY INTO

-- Correct: use $1 column references (Parquet columns by position) COPY INTO parquet_col FROM ( SELECT $1:o_custkey::number, $1:o_orderdate::date, $1:o_orderstatus::varchar, $1:o_totalprice::varchar FROM @mystage/mydata.parquet );

Named column refs like o_custkey::number without $1 is wrong for Parquet.

Domain 4 — Snowsight & Dashboards

High Weight

Chart Types

  • Scatter plot → identify natural clusters
  • Heat grid → show product transfer routes / frequency matrix
  • Line chart → time-series trends
  • Bar chart → category comparisons
  • Scorecard tile → single KPI with % change vs comparison

Dashboard Filters

  • Custom filter → referenced as :filter_name in queries
  • System date filter (:daterange) → use WHERE col = :daterange
  • System bucket filter (:datebucket) → use in GROUP BY
  • Filter runs using role of user who CREATED the filter
  • "SQL keyword" in filter = filter name inserted into query

Scorecard Tile

  • Shows VALUE (large number) + % change vs comparison
  • Tile: 38 ↑ 12% → 38 = company avg, 12% = percent difference from industry avg (34)
  • (38-34)/34 × 100 = 11.76% ≈ 12%

Worksheet → Dashboard

  • Move worksheets into dashboard (least operational overhead)
  • When creating dashboard from shared worksheet → original users LOSE access; permissions revoked
  • Shared worksheets: multiple users running same time creates new versions
  • Collaborators can see real-time edits (D); version history preserved

Column Histogram

  • Click column header in results → see distribution histogram without re-running
  • WIDTH_BUCKET() — manual bucketing function
  • Use "Go to Chart → histogram" requires two variables (bivariate)

Aggregate Functions in Charts

  • Snowsight chart buckets support: Any Value, Average, Max (select THREE)
  • NOT supported: Mode, HLL, Standard Deviation (for buckets)
💡

Filter syntax: WHERE start_time = :daterange links a tile to the system date filter. :date_filter is a custom filter name. SYSDATE() / CURRENT_TIMESTAMP() are hardcoded, not dynamic dashboard filters.

Domain 5 — Query Performance & Optimization

Medium Weight

Query Profile Reading Key Signals

  • High % in Rescan node → exploding join (cartesian-like result)
  • Spilling indicator → query spilling (memory overflow)
  • Inefficient pruning → micro-partition scan too wide
  • Join [S] → small table; large % in Rescan = exploding join

Spilling Fix — No Cost Increase Optimization

  • Create materialized view clustered on date column on the spilling table
  • Run against zero-copy clones → avoids contention only
  • External tables → custom partitioning, not spill fix
  • Split into CTEs with temp tables → helps but adds cost

USE_CACHED_RESULT = FALSE Slow Query Reason

Two identical queries on same-size tables — Query 2 runs slower because USE_CACHED_RESULT was set to FALSE, bypassing the result cache. Otherwise same-query results are served from cache instantly.

Result Cache Access RESULT_SCAN

-- Access results of previous query by its ID SELECT * FROM TABLE(RESULT_SCAN(LAST_QUERY_ID())); -- LAST_QUERY_ID(-1) = 2 queries back -- desc result last_query_id() = WRONG syntax

Query Acceleration Service SYSTEM$ESTIMATE_QUERY_ACCELERATION

Returns JSON with estimatedQueryTimes: keys 1, 2, 4, 8 represent scale factors (how much to scale the warehouse). Not micro-partitions, not clusters.

Cardinality Estimation Large Tables

  • HLL(*) — HyperLogLog, fast approximate distinct count for trillion-row tables
  • APPROX_COUNT_DISTINCT — alias for HLL behavior
  • COUNT(DISTINCT *) — exact but expensive on huge tables
  • SYSTEM$ESTIMATE — not a real function for cardinality

Domain 6 — Data Sharing, Marketplace & Secure Views

Medium Weight

Secure Views — Benefits Security

  • Hides query definition from consumers
  • Users CANNOT observe quantity of underlying data (option A ✓)
  • Amount of data scanned and total volume are obfuscated
  • Secure views are LESS performant than regular views (optimizer restricted)

Data Consumer Permissions Sharing

  • ✓ Copy data from shared DB into consumer account
  • ✓ Create streams on shared DB objects
  • ✓ Grant shared DB roles to consumer roles
  • ✗ Create clone of shared database
  • ✗ Use Time Travel on shared DB
  • ✗ Edit comments in shared DB

Cross-Account Row Access Policy Sharing + Security

When sharing views across accounts, use CURRENT_ACCOUNT() in the row access policy to give consumers access to rows belonging to their account. Not CURRENT_ROLE, not CURRENT_USER.

Marketplace Cross-DB View Scenario

Need to join ADDRESS_DATA (Marketplace DB) with GLOBAL_DWH.ORDERS table → create the view in the GLOBAL_DWH database (where ORDERS lives). You can cross-reference ADDRESS_DATA from there. Cannot create view in ADDRESS_DATA referencing GLOBAL_DWH objects without write access.

Domain 7 — Stored Procedures, UDFs & Tasks

Medium Weight

Stored Proc vs UDF Key Difference

  • Stored Proc → dynamic SQL, procedural logic, modular code encapsulation ✓
  • UDF → called inside SELECT, returns scalar/table value
  • Dynamic SQL (EXECUTE IMMEDIATE) → only Stored Procedures
  • UDFs cannot dynamically CREATE/DROP tables

Nested Function Call Tracing Logic

-- func2(10) → select p * 10 → 100 -- func1(100) → select v * 10 → 1000... wait -- func2(10): p=10 → p*10 = 100 -- func1(100): v=100 → v*10 = 1000... -- proc1(1000): result=1000 >= 30 → return 1000-10 = 990? -- Trace carefully: result >= 30 → result-10

Always trace inside-out: innermost function first.

Task with Stream (Event-Driven) Correct Syntax

-- CORRECT: WHEN clause triggers on new stream data, no SCHEDULE CREATE TASK mytask1 WAREHOUSE = mywh WHEN SYSTEM$STREAM_HAS_DATA('MYSTREAM') AS ... -- WRONG: SCHEDULE + WHEN together doesn't auto-trigger on stream alone -- WRONG: SCHEDULE = 'STREAM' is not valid syntax

DAG Task Retry EXECUTE TASK

ETLDAILY (DAG root) failed at STEP10 → STEP11, STEP12 didn't run. To re-run STEP10 and its dependents: EXECUTE TASK STEP10 RETRY LAST; — not the root ETLDAILY, not ALTER TASK RESUME.

Domain 8 — Time Travel, Zero-Copy Cloning & Storage

Low-Medium Weight

Zero-Copy Clone for Investigation Scenario

PROD DB has 15TB TRANS_HIST (2001–2022). Need a copy in DEV to investigate 2019 anomaly without extra storage cost:

-- CORRECT: Clone (no storage cost for unchanged data) CREATE TABLE DEV.PUBLIC.TRANS_HIST CLONE PROD.PUBLIC.TRANS_HIST; -- WRONG: CREATE TABLE … LIKE (only copies structure, no data) -- WRONG: CREATE TABLE … AS SELECT * (full copy = extra storage) -- WRONG: CREATE TABLE … AS SELECT WHERE year=2019 (partial only)

Domain 9 — ML Functions & Forecasting

Low Weight

modelX.FORECAST → Save Results Workflow

  • After calling modelX!FORECAST(…), results land in a RESULT_SCAN
  • To save: CREATE TABLE tableX AS SELECT * FROM TABLE(RESULT_SCAN(LAST_QUERY_ID()))
  • Cannot load directly INTO tableX as function argument
  • Cannot pass the new table as a function argument

Sales Prediction Function Which Function?

To predict sales revenue through end of fiscal year → use REGR_SLOPE (gives slope of linear regression — rate of change, used for projection). Not CORR (correlation only), not COVAR_SAMP (covariance), not VARIANCE.

Domain 10 — Access Control & SSO

Low Weight

SSO Setup Role Role Required

BI platform accessing Snowflake via SSO → setup requires SECURITYADMIN role. Not ACCOUNTADMIN, not USERADMIN, not SYSADMIN.

ACCOUNT_USAGE vs INFORMATION_SCHEMA Monitoring

  • ACCOUNT_USAGE → up to 365 days history, 45-min latency
  • INFORMATION_SCHEMA → real-time, 7-day history only
  • Dashboard query failing: DB1.ACCOUNT_USAGE does not exist → must use SNOWFLAKE.ACCOUNT_USAGE (change DB context to SNOWFLAKE)

Troubleshoot Dashboard Filter Failures Alert History

Daily dashboard filter refresh failing → query SNOWFLAKE.ACCOUNT_USAGE.ALERT_HISTORY. Not LOGIN_HISTORY, not ACCESS_HISTORY, not TASK_HISTORY.

Domain 12 — TABLE GENERATOR & Approximate Functions

Low Weight

TABLE(GENERATOR) Row Count Logic

SELECT seq4(), uniform(1, 10, RANDOM(12)) FROM TABLE(GENERATOR(TIMELIMIT => NULL)) ORDER BY 1;

TIMELIMIT => NULL means unlimited time but uniform(1,10,...) constrains row generation — answer depends on ROWCOUNT, not TIMELIMIT. With TIMELIMIT=NULL and no ROWCOUNT = 0 rows (NULL timelimit and no rowcount = no generation).

Similarity Estimation MinHash

  • MinHash → estimates approximate similarity between two or more datasets (Jaccard similarity)
  • HyperLogLog → cardinality (distinct count) estimation
  • APPROX_PERCENTILE → percentile estimation
  • APPROX_TOP_K → most frequent values

Domain 13 — Dimensional Modelling

Low Weight

Why Dimensional Model over Flat Table? (Select TWO) BI Design

  • ✓ A — Dimensional modelling improves query performance over a single table
  • ✓ C — Combining facts and dimensions in a single flat table limits scalability and flexibility
  • ✗ B — Dimensional modelling does NOT save storage (it's denormalized into dims, more storage)
  • ✗ D — Ad-hoc analyses benefit from both models
  • ✗ E — Snowflake does not inherently perform better with dimensional models

Foreign Key After Table Rename DDL Behaviour

Table "mother" renamed to "father" via ALTER TABLE RENAME — FK on "son" that references "mother" is automatically updated to reference the renamed table (D). Snowflake tracks object identity, not name strings.
